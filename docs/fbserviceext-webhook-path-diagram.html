FBServiceExt Webhook Path Diagram

Current end-to-end path after the stateless voting refactor: fast API ACK, queue-based background flow, runtime ActiveShowId gate, signed payloads, show-scoped cooldown and configurable NormalizedEvents persistence.

1. API ACK lane

Meta → API Incoming webhook POST
May contain GET_STARTED, #forgetme, candidate postback, confirmation postback or garbage traffic.
↓
Controller Read body + inspect + signature validation
  • body read
  • payload classification
  • Meta signature validation
↓
Early gates Drop fast when possible
  • oversized payload
  • garbage traffic
  • voting-disabled regular traffic
↓
Raw publish AcceptAsync + RabbitMQ raw ingress
Once the raw envelope is published, API returns HTTP 200.

2. Background worker lane

Raw worker Normalize raw ingress
Creates business-friendly normalized events from raw webhook envelopes.
↓
Business filter Keep only relevant workflow events
  • GET_STARTED
  • #forgetme
  • postback payloads
↓
Normalized queue Publish to fbserviceext.normalized.events
Processor workers consume business events from here.
↓
Processor Stateless vote / erase workflow
  • GET_STARTED → runtime gate → cooldown gate → send signed VOTE1 carousel
  • VOTE1 → validate signature + show → send CONFIRM1
  • CONFIRM1 → validate signature + show + issuedAt + action → accept vote or recover
  • #forgetme → send signed ERASE1 prompt
  • ERASE1 → full erase across DB/Redis/cache

3. Runtime state and persistence

Runtime state Operator-controlled gate
  • VotingStarted
  • ActiveShowId
  • candidate list from config
Secrets Signed payload validation
  • VotingWorkflow:PayloadSignatureSecret
  • DataErasure:ConfirmationPayloadSecret
Redis state What still lives in Redis
  • show-scoped cooldown key
  • runtime gate values
  • user account name cache
SQL + outbound Persistence and external effects
  • AcceptedVotes with ShowId + UserAccountName
  • NormalizedEvents with mode Full / Minimal / Disabled
  • Meta outbound or fake-meta store delivery
The key architectural distinction now is simple: the API host only admits and publishes webhook traffic, while actual voting logic lives entirely in background workers. No vote session snapshots remain; correctness comes from signed payloads, ActiveShowId, cooldown keys, and final writes to AcceptedVotes.
