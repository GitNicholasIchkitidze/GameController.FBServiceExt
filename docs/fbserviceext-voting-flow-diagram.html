FBServiceExt Voting Business Flow

Readable operator view of the current vote path: GET_STARTED opens candidate selection, the next tap creates a confirmation challenge with the correct button randomized, and only the correct confirmation inside the time window creates a persisted vote and starts cooldown.

What This Diagram Shows

Candidate selection itself is not a vote. The business effect happens only after the confirmation challenge succeeds. This is the core anti-auto-clicker mechanism in the current flow.

Current Runtime Shape

Content, timeouts, cooldown, carousel cards, and button labels are all config-driven. Messenger messages are sent from the worker through MetaMessengerClient.

What actually counts as a vote: the candidate tap from the carousel is only a selection. The business vote is created only after the confirmation challenge succeeds inside the timeout window.
Countdown nuance: Messenger generic/button templates can show the configured 120 seconds as static text, but they do not provide a native live ticking countdown bubble. Real enforcement still happens in backend session expiration and cooldown logic.
