FBServiceExt Webhook ACK Latency Breakdown

Updated for the current architecture: payload inspection, request-level garbage and voting gates, a pub/sub-backed Redis voting snapshot, and pooled RabbitMQ raw publish all shape the path before HTTP 200. Some requests now short-circuit early, and some requests skip the voting gate lookup entirely, but business-relevant traffic still pays the publish cost before the ACK returns.

Core Reading

The ACK latency story is still mostly about stage 5. The earlier stages now do less work on the hot path: garbage requests can stop before queue publish, voting-disabled drops can come from an in-memory snapshot, and non-droppable traffic can skip the gate lookup entirely. Real business traffic still waits for RabbitMQ publish before the ACK returns.

Validated Baseline

500 incoming events/sec for 5 minutes passed in the Performance profile, including EventsPerRequest = 1 / 2 / 3. That baseline is what this latency model now reflects.

What This Means in Practice
Operational interpretation of the ACK path

Good

Sub-300 ms ACK remains a healthy target for this architecture. If your request returns in this zone, Meta retries are unlikely and the system is operating with comfortable headroom.

Watch

If p95 or p99 starts climbing, the first awaited place to inspect is stage 5, not the background workers. Stage 4 now removes more waste than before, but it cannot eliminate the RabbitMQ hop for real business traffic.

Bad

Multi-second ACKs are dangerous because retries and duplicate deliveries become likely. When that happens, publish-path contention or broker pressure usually surfaces before background business throughput becomes the visible issue.

Why await still matters: await _webhookIngressService.AcceptAsync(...); return Ok(); means the response is returned only after the raw publish path finishes. Async is non-blocking for the thread, but it still delays the HTTP response until the awaited work completes.
What changed from the earlier version: payload inspection and the VotingStarted gate now sit before RabbitMQ, the gate is served from a pub/sub-backed in-memory snapshot, non-droppable traffic skips that lookup, and unused request-header copying was removed from the raw ingress envelope. These changes make the ACK path cheaper, but they do not remove the raw publish step from real business traffic.
